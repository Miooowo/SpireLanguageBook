杀戮尖塔卡牌描述规范与指南

为《杀戮尖塔 2》（STS2）中卡牌、遗物、能力、药水及事件等描述文本的撰写提供统一规范。 无论你是制作 Mod、设计自定义内容，还是想弄清官方文案怎么写——本文档都能作为参考。 描述须使用游戏规定的标记语法；在此基础上，用中文把效果写清楚即可。

01.普适原则

  • 机制优先，文字服从玩法。先确定这张牌/遗物实际做什么，再写描述；不得写出与代码逻辑不符的句子。
  • 清晰优先于文采。描述的首要目标是让玩家在战斗中一眼读懂效果；事件叙事可适当润色，但不可模糊机制含义。
  • 符合中文表达习惯。避免拖沓长句；效果较多时宜拆句，并用换行分隔。
    • 欠佳：「获得 X 点格挡，然后升级你手牌中的一张牌。」
    • 推荐：「获得 X 点[gold]格挡[/gold]。\n[gold]升级[/gold]你[gold]手牌[/gold]中的一张牌。」
  • 语气与内容类型匹配。卡牌描述用简洁说明体；事件文本可更叙事化，但不要偏离机制表述。
  • 同类效果句式统一。伤害、格挡、抽牌、给予状态等常见模板，建议与官方范例保持一致（见第 08 节）。

02.描述数据文件结构

游戏中的描述文本存放在 zhs/ 目录下的 JSON 文件中（Mod 亦多采用相同格式），键名格式为 实体ID.字段名。

文件常用字段说明
cards.jsontitle、description卡牌名称与效果描述
relics.jsontitle、description、flavor、eventDescription遗物效果、风味、事件内描述
powers.jsontitle、description、smartDescription战斗内能力；smart 为带数值的动态版
potions.jsontitle、description药水效果
card_keywords.jsontitle、description虚无、消耗、固有等关键词释义
enchantments.jsondescription、extraCardText附魔说明与卡牌上的简短附加行
events.jsonpages.*.description、options.*事件叙事与选项文本
static_hover_tips.jsontitle、description界面悬停提示
撰写时只修改字符串值（即描述正文），不要改动 JSON 键名。键名与游戏内 ID 绑定，改动会导致加载失败。

03.标记语法（撰写时必须遵守）

游戏使用一套富文本标记语言。撰写描述时须正确使用下列结构；你可以自由书写的是标签内的中文，以及标签之间的连接词句。

3.1 颜色与格式标签

成对标签：[标签名]文字[/标签名]。可嵌套。

3.2 变量占位符

花括号块：{变量名} 或 {变量名:修饰符}。修饰符、冒号、括号均为语法的一部分。

3.3 分支与条件

竖线 | 分隔不同分支；常见于 :show:、:cond:、:choose() 等修饰符中。

标记文本（写入 JSON 的字符串）
游戏内预览
切勿将 {Damage:diff()} 预填为具体数字（如「造成 6 点伤害」）。数值由游戏运行时注入，预填会导致升级后不更新。

04.颜色与格式标签

标签用途典型场景
[gold]金色关键词格挡、力量、牌堆名、机制动词（升级、消耗、生成）
[blue]蓝色数值遗物/药水中的固定数字，或包裹变量
[green]绿色数值生命回复、最大生命等增益数值
[red]红色警告伤害、诅咒、未公开内容占位
[purple]紫色特殊附魔名（锋利、墨影、迅捷等）
[orange] [aqua]叙事强调主要用于事件文本中的 NPC、地点
[sine] [jitter] [b] [rainbow]动画/强调事件叙事专用；卡牌描述中极少使用

4.1 [gold] 关键词

凡游戏内已有的机制词、牌堆名、状态名，均用 [gold] 包裹，以便玩家点击关键词查看悬停说明。

05.变量与修饰符

修饰符含义撰写时注意
:diff()动态数值（含升级差值着色）保留修饰符；中文写在变量前后均可，但全库应统一
:energyIcons()渲染为能量球图标写作「获得{Energy:energyIcons()}」而非「获得 1 点能量」
:starIcons()渲染为辉星 ★储君机制专用
:show:A|B升级与否显示不同文本两个分支都要写全；竖线不可省略
:cond:A|B条件分支分支内可含嵌套标签与变量
:choose(Type):A|B枚举分支choose(...) 内为程序枚举，保持原样
:plural:A|B复数/量词分支中文常两分支相同，但仍须保留结构
{InCombat:…|}战斗中才显示的附加行空分支 |} 表示非战斗时隐藏

06.标点符号与换行

  • 中文描述使用中文全角标点：。，、；：？！
  • 句末统一使用句号 。，即使是关键词释义（card_keywords）也应完整成句。
  • 并列的独立效果之间使用换行 \n 分隔，优于分号堆砌。
    • 规范：「获得 {Block:diff()} 点[gold]格挡[/gold]。\n抽 2 张牌。」
  • 保留变量与标签紧邻的写法，标签两侧一般不需要额外空格。
  • 变量名、修饰符使用半角英文字符，与程序绑定，不可改成中文。

6.1 量词

遵循「数词 + 量词 + 名词」习惯，数词来自变量时放在变量之后：

  • 造成{Damage:diff()}点伤害
  • 给予{VulnerablePower:diff()}层[gold]易伤[/gold]
  • 抽{Cards:diff()}张牌
  • 获得{Block:diff()}点[gold]格挡[/gold]

8.1 多效果排序

一张牌有多个效果时，描述中的书写顺序应与该牌的实际结算顺序一致：先结算的写在前，后结算的写在后；对应到卡牌上会呈现为从上至下阅读。

  • 这不是固定模板——不同卡牌结算顺序不同，须以该牌的实际逻辑为准。
  • 撰写 Mod 卡牌时，先确认代码中的效果触发顺序，再按此顺序组织描述文本。
  • 每个效果仍建议单独成句，用换行 \n 分隔，便于玩家逐项阅读。
不要套用「先伤害后格挡」之类的通用顺序；若结算顺序与描述顺序不一致，玩家按文本操作时会产生误解。

09.遗物、能力与药水

9.1 遗物 description 与 flavor

  • description：机制说明，句式与卡牌类似，常用 [blue]{变量}[/blue]。
  • flavor：风味文本，可更文学化；未公开内容保持官方占位句式。
  • eventDescription：事件选项中的简短效果，须与 description 语义一致。

9.2 能力 description 与 smartDescription

  • description：通用说明，常不含具体层数。
  • smartDescription：战斗内悬停时显示，含 [blue]{Amount}[/blue] 等动态数。
  • 两者机制措辞须一致，仅数值详略不同。
description
smartDescription

9.3 药水

多为一次性效果，描述简短，常用 [blue]{变量}[/blue] + [gold]关键词[/gold]。消耗类药水须写清目标（如「所有敌人」）。

10.事件与叙事文本

  • 事件正文可使用 [sine]、[jitter]、[rainbow] 等动画标签；卡牌描述中不要滥用。
  • 选项 options.*.description 写机制后果，宜短；页面 pages.*.description 写叙事，可长。
  • 选项描述中的数值效果仍须遵循第 04–06 节标记规范。
  • loss 字段为死亡叙事，可含 {character}、{event} 等变量，变量名不可改动。

11.常见错误对照

错误说明正确做法
删除或重命名变量 {Damage} → 写成「6」 保留 {Damage:diff()}
修改变量名 {Block:diff()} → {格挡:diff()} 变量名须与程序字段一致，保持英文
遗漏 gold 标签 手牌、弃牌堆等未高亮 机制词加 [gold]
卡牌数值误用 [blue] [blue]{Damage:diff()}[/blue] 卡牌用 {Damage:diff()} 即可
破坏 :show: 分支 只写一个分支、删除竖线 保留 A|B 完整结构
JSON 非法转义 换行未写成 \n 在 JSON 字符串内用 \n 表示换行
术语前后不一 同一牌堆「抽牌库」「抽牌堆」混用 在范例库中检索，统一用词
